# 技术调研 Skill

用于指导 AI 完成系统化、高质量的技术调研任务，输出可直接用于决策的调研报告。

## 调研流程总览

```
明确需求 → 拆解问题 → 信息收集 → 方案分析 → POC验证（可选）→ 结论 → 报告
```

---

## 第一步：明确需求与背景

在开始调研前，先从对话中提取或向用户确认以下信息：

1. **调研目标**：要解决什么问题？现有方案的痛点是什么？
2. **业务背景**：项目规模、团队技术栈、预期用户量、性能要求
3. **约束条件**：时间、成本、开源/商业、语言/平台限制
4. **决策粒度**：是技术方向选择、具体框架选型，还是特定功能实现方式？
5. **已知信息**：用户已了解哪些方案？倾向于哪个方向？

> 为什么先做这步：错误理解需求是调研失败的头号原因。宁可多问一个问题，也不要做完才发现方向跑偏。

---

## 第二步：拆解调研维度

根据需求，确定需要比较的维度。常见维度（按场景选取，不必全选）：

**技术能力维度**
- 核心功能覆盖度（是否满足需求）
- 性能（吞吐量、延迟、内存占用）
- 扩展性（水平扩展、插件机制）
- 稳定性（成熟度、大规模生产案例）

**工程质量维度**
- 文档质量与完整度
- 社区活跃度（GitHub Stars 趋势、Issue 响应速度、近期 commit）
- 学习曲线与上手难度
- 测试覆盖率、CI/CD 友好程度

**运维与集成维度**
- 与现有技术栈的兼容性
- 部署复杂度
- 监控、告警、可观测性支持
- 升级维护成本

**商业与风险维度**
- 开源协议（MIT/Apache/GPL 对商用的影响）
- 背后维护方（个人/公司/基金会）
- 长期维护风险（是否有被放弃的迹象）
- 安全漏洞历史

---

## 第三步：信息收集

按优先级收集信息，先快速过一遍，再深入关键点：

**第一轮（快速）**
- 官方文档首页、README、特性列表
- GitHub 仓库：Stars、最近 commit 时间、open issues 数量
- 官方 benchmark（如有）

**第二轮（深入）**
- 官方文档的架构设计章节
- CHANGELOG / Release Notes（了解演进方向）
- 已知问题和局限（官方 issue、社区讨论）
- 真实用户案例（生产环境使用报告、技术博客）
- 对比文章（搜索 "A vs B"，注意文章时效性）

**搜索策略**
- 搜索 `[技术名] production experience`、`[技术名] limitations`、`[技术名] vs [竞品]`
- 查 GitHub Issues 的 `won't fix` 和 `known limitation` 标签
- 查近期（6个月内）的技术会议分享或博客文章

> 注意信息时效性：技术发展快，2年前的对比文章可能已经过时。优先参考近12个月的内容。

---

## 第四步：方案对比分析

将收集到的信息结构化为对比矩阵，每个方案按相同维度评估：

**评估格式**
```
方案名称
- 核心优势：（2-3条，具体说明为什么是优势）
- 主要局限：（2-3条，不要回避缺点）
- 适用场景：（什么情况下用这个最合适）
- 不适用场景：（什么情况下不推荐）
- 社区/生态：（活跃程度、周边工具）
- 与用户技术栈的契合度：（具体说明）
```

评估要诚实：对每个方案的缺点都要直接说出来，不要因为某个方案"知名度高"就回避问题。

---

## 第五步：POC 验证（当不确定性高时）

如果存在以下情况，建议做 POC：
- 技术方案对项目有决定性影响，且存在关键不确定因素
- 官方文档声称支持某特性，但无法确认在具体场景下是否可行
- 两个方案文档上看起来差不多，需要实际数据区分

**POC 范围原则**：最小化验证目标，只验证不确定的核心点，不要做成完整原型。

**POC 验证清单**
- [ ] 定义验证什么（一句话说清楚要证明/证伪什么假设）
- [ ] 搭建最小化环境
- [ ] 运行测试（包括正常路径和边界条件）
- [ ] 记录数据（性能数字要有环境信息：CPU/内存/数据量）
- [ ] 得出明确结论（可行/不可行/有条件可行）

---

## 第六步：形成结论

结论必须给出明确建议，不能模棱两可：

**结论结构**
1. **推荐方案**：直接说推荐哪个，以及核心理由（1-2句）
2. **备选方案**：在什么情况下考虑备选（给出具体条件）
3. **风险提示**：推荐方案的已知风险和应对措施
4. **下一步行动**：如果采纳推荐，接下来应该做什么

> 好的结论是敢于给出明确建议的结论。"两个方案各有优缺点，需要根据实际情况选择"——这不是结论，这是废话。

---

## 输出报告格式

完成调研后，生成结构化报告。详细格式见 `references/report-template.md`。

报告用中文，必须包含：
- 执行摘要（结论先行，让读者第一段就知道推荐什么）
- 调研背景与目标
- 各方案详细分析
- 横向对比（用表格）
- 明确结论与建议
- 参考资料（附上调研时参考的关键链接）

---

## 注意事项

- **时间管理**：调研本质是工作，不是学习。要聚焦在"够用"的信息上，不要沉迷于深度学习某项技术
- **阶段性沟通**：调研进行到关键节点时，与需求方同步进展，防止方向偏差
- **区分事实与观点**：在报告中明确哪些是文档/官方说明（事实），哪些是基于分析的判断（观点）
- **不做"尚无结论"的调研**：调研结束必须有明确输出，哪怕结论是"目前没有合适方案，建议自研/暂缓"
