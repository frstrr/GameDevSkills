# 搭建 AGENTS.md 技能

为项目创建 AGENTS.md（AI Agent 的 README）及配套子系统文档，让 AI 编码代理在首次接触项目时就能高效、规范地工作。

## 什么是 AGENTS.md

AGENTS.md 是一个行业开放标准，放在项目根目录，作为所有 AI 编码代理（Cursor、Copilot、Warp、Windsurf、Claude Code 等）的全局指令文件。它相当于"给 AI 看的 README"。

> Claude Code 使用 `CLAUDE.md`，可通过符号链接保持同步：`mklink CLAUDE.md AGENTS.md`（Windows）或 `ln -s AGENTS.md CLAUDE.md`（Unix）。

---

## 执行流程

```
1. 了解项目 → 2. 创建 AGENTS.md → 3. 创建子系统文档 → 4. 验证 → 5. 提交
```

---

## 第一步：了解项目信息

在创建之前，先收集以下信息（从项目中读取或向用户确认）：

1. **项目简介**：一句话描述项目是什么
2. **技术栈**：引擎/语言/框架版本、关键依赖库
3. **构建与测试命令**：如何编译、运行、测试
4. **目录结构**：核心目录的用途（描述能力而非罗列路径）
5. **编码规范**：命名约定、代码风格、架构模式
6. **已有规范文件**：是否已存在 `.cursorrules`、`.clinerules` 等，有则作为参考
7. **子系统划分**：项目有哪些独立子系统需要分别出文档

**信息收集方式**：
- 读取 README.md、package.json / .csproj / build.gradle 等项目配置文件
- 扫描目录结构（`ls` 或 `tree`）
- 查看现有的 lint/format/test 配置
- 读取已有的 AI 规则文件（.cursorrules 等）

---

## 第二步：创建根目录 AGENTS.md

在项目根目录创建 `AGENTS.md`，严格遵守以下原则：

### 内容原则

**必须包含（六大核心区域）**：
1. **项目简介** — 一句话说明项目是什么
2. **技术栈** — 具体到版本号，如 "Unity 2022.3 LTS / C# / UniTask 2.5"
3. **构建与测试命令** — 精确命令，带参数和选项
4. **目录结构概览** — 描述能力而非罗列每个文件路径
5. **编码规范** — 命名、风格、架构模式，附 1-2 个代码范式
6. **禁止事项** — 绝对不能做的事，用三级边界（始终做 / 先问 / 绝不做）

**可选包含**：
- Git 工作流（分支策略、提交规范、PR 要求）
- 安全事项（API 密钥处理、敏感数据）
- 指向子系统文档的引用

**不应包含**：
- 详细的功能设计（放在子系统文档里）
- 大量代码示例（膨胀上下文，1-2 个范式足够）
- 频繁变动的文件路径（用能力描述代替路径）
- 已在 README.md 中充分说明的内容（避免重复）

### 格式原则

- 保持精简，**根 AGENTS.md 建议不超过 150 行**
- 命令放在前面，Agent 会频繁引用
- 用代码示例代替文字描述，一个真实代码片段胜过三段文字
- 使用渐进式披露：根文档只放全局规则，详细内容引用子文档
- 把 AGENTS.md 当作"活文档"，随项目演进持续更新

### 三级权限边界

```markdown
## 权限边界

### 始终可以（无需确认）
- 读取文件、列出目录
- 运行 lint / format / 单个文件的类型检查
- 运行单个测试

### 先问再做
- 安装新依赖
- 删除文件
- 运行完整构建或端到端测试
- git push

### 绝对禁止
- 提交密钥/凭证
- 修改生产环境配置
- 在代码中硬编码敏感信息
```

### 参考模板

完整模板见 `references/agents-md-template.md`，可直接复制到项目根目录后填写。

## 第三步：创建子系统文档（按需）

如果项目有多个独立子系统，为每个子系统创建专属的 AI 开发文档。

### 子系统文档位置

两种方式（根据项目结构选择）：

**方式一：集中管理**（推荐中小项目）
```
项目根目录/
├── AGENTS.md           ← 全局指令
└── doc/ai/
    ├── 01-角色系统.md
    ├── 02-战斗系统.md
    └── 03-UI系统.md
```

**方式二：分布式管理**（推荐大型项目/monorepo）
```
项目根目录/
├── AGENTS.md           ← 全局指令
├── src/player/
│   └── AGENTS.md       ← 角色子系统指令
├── src/combat/
│   └── AGENTS.md       ← 战斗子系统指令
└── src/ui/
    └── AGENTS.md       ← UI 子系统指令
```

> 子目录的 AGENTS.md 会与根目录的合并，且子目录优先级更高。

### 子系统文档结构

每份子系统文档应包含以下标准章节：

```
# 系统名称
> Agent角色 | 关联文档 | 技术栈

## 角色定义（告诉 AI 它扮演什么角色）
## 全局约束（引用 AGENTS.md 或写子系统特有约束）
## 功能需求（具体要做什么，用数值和代码代替形容词）
## 接口定义（与其他系统的边界、调用方式）
## 验收标准（可验证的完成条件）
## 约束与禁止（子系统级别的禁止事项）
## 任务清单（拆解为可验证的小步骤）
```

完整模板见 `references/subsystem-doc-template.md`。

### 关键原则

- **模块化**：一个 Agent 一份聚焦文档，Agent 只读自己负责的文档 + AGENTS.md
- **具体化**：用数值代替形容词（❌ "间距随难度增大" ✅ "0-100米：间距2.0~4.0"）
- **可验证**：每个任务都有明确的验收标准
- **边界清晰**：明确禁止修改其他系统的代码
- **任务粒度**：每个任务应是一个 Agent 在单次会话中能完成的量
- **单文档不超过 500 行**，避免超出 AI 上下文窗口

## 第四步：验证文档质量

创建完成后，对照以下清单自检：

### 根 AGENTS.md 自检
- [ ] 有一句话项目简介
- [ ] 技术栈写明了具体版本号
- [ ] 构建/测试命令精确，可直接复制执行
- [ ] 目录结构描述能力而非罗列路径
- [ ] 有编码规范和代码范式（至少1个）
- [ ] 有三级权限边界（始终可以 / 先问 / 绝不做）
- [ ] 有指向子系统文档的引用（如有子系统）
- [ ] 总行数不超过 150 行

### 子系统文档自检
- [ ] 有明确的 Agent 角色定义
- [ ] 有全局约束或引用 AGENTS.md
- [ ] 需求具体，有数值和条件
- [ ] 有接口定义（与其他系统的边界）
- [ ] 每个任务有可验证的验收标准
- [ ] 有约束与禁止
- [ ] 任务粒度适合单次会话
- [ ] 包含代码范式（至少1个关键模式）
- [ ] 单文档不超过 500 行

---

## 第五步：提交与维护

1. **提交到版本控制**：AGENTS.md 和子系统文档一起提交
2. **CR 审查**：像审查代码一样审查 AGENTS.md 的变更
3. **持续迭代**：Agent 犯错时 → 更新文档的"约束与禁止"章节，防止重犯
4. **保持同步**：构建命令、目录结构变化时，同步更新 AGENTS.md

> 最好的 AGENTS.md 是通过迭代成长的，不是一次性写完的。每次 AI 犯错，就把修正规则加入文档。

---

## 注意事项

- **精简优先**：AI 大模型能稳定遵循约 150-200 条指令，内容越多遵守率越低
- **描述能力而非路径**：文件路径经常变，写"认证模块"比写"src/auth/handlers.ts"更稳定
- **不要自动生成**：手动编写，优先克制而非全面覆盖
- **代码示例 > 文字描述**：1个代码范式胜过3段描述
- **禁止事项很重要**：告诉 AI 不能做什么，比告诉它要做什么更关键
- **多工具兼容**：一份 AGENTS.md 可服务于 Cursor、Copilot、Warp 等所有工具，避免为每个工具维护单独配置文件
